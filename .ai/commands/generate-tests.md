# generate-tests.md

**Command:** `generate-tests`  
**Version:** 1.0  
**Library:** `.ai/commands/`  
**Applies to:** All PIs, all sprints

---

## Purpose

Use this command to generate the full test suite for a completed implementation.

This command produces unit tests, integration tests, contract validation tests, and acceptance tests. It does not generate code-under-test. The implementation must exist before tests are generated.

Tests generated by this command must be real — they must actually fail when the feature is broken.

---

| Input | Location | Required |

## Execution Steps

### Step 1 — Identify what must be tested

Read all implementation files in `{target_folder}`. Categorise every testable unit:

```
Domain layer functions:     list every public method in domain/
API layer endpoints:        list every HTTP route
Kafka consumers:            list every consumer function
Kafka producers:            list every producer call
Database operations:        list every repository method
Cross-tenant operations:    list every operation that filters by tenant_id
Contract validations:       list every place a contract is validated
```

### Step 2 — Map acceptance criteria to tests

For every acceptance criterion in `ACCEPTANCE_CRITERIA.md` for `{story_id}`:
- Write one test that directly mirrors the Given/When/Then
- Test name must reference the criterion ID: `test_ac_{criterion_id}_{description}`
- The test must fail if the criterion is not satisfied

### Step 3 — Write unit tests

Location: `src/tests/unit/{service_name}/`

Rules:
- One test file per source module: `test_{module_name}.py`
- All external dependencies mocked — no real DB, Kafka, or Redis in unit tests
- Use `pytest` fixtures for common setup
- Each test function tests exactly one behaviour
- Naming: `test_{method_name}_{scenario}` e.g. `test_resolve_returns_agent_for_capability`

Required unit test coverage:
- Every public method in `domain/` — happy path + at least two edge cases
- Every error path — what happens when a dependency is unavailable
- Every validation — invalid input rejected with correct error type

### Step 4 — Write integration tests

Location: `src/tests/integration/{service_name}/`

Rules:
- Use Testcontainers for all external dependencies (PostgreSQL, Redis, Kafka)
- Test the full call path from API → domain → infrastructure → external
- Never mock the service under test — only mock services it calls that are not under test

Required integration tests:
- Primary happy path end-to-end
- Kafka event published with correct envelope when operation succeeds
- Kafka event published to DLQ when operation fails after retries
- Database operation respects RLS — cross-tenant query returns 0 rows

### Step 5 — Write contract validation tests

If the story registers an agent or tool, or publishes events:

**Agent contract test:**
```python
def test_agent_contract_valid():
    registration = load_json("src/agents/{agent}/registration.json")
    schema = load_json("contracts/agent-contract.schema.json")
    validate(registration, schema)  # must not raise
```

**Event envelope test:**
```python
def test_published_event_conforms_to_envelope():
    event = capture_published_event(topic="aep.task.created")
    schema = load_json("contracts/event-envelope.schema.json")
    validate(event, schema)  # must not raise
```

### Step 6 — Write idempotency tests (agents and tools only)

If the story implements an agent or tool operation:

```python
def test_retry_does_not_duplicate_side_effect():
    # Execute the operation twice with the same idempotency key
    result_1 = await agent.run(context_with_idempotency_key)
    result_2 = await agent.run(context_with_idempotency_key)
    # Assert: only one side effect produced (e.g., one PR created, one row inserted)
    assert side_effect_count() == 1
```

### Step 7 — Write security tests (if story touches auth, data, or credentials)

Required when the story involves authentication, data access, or secrets:

- Cross-tenant isolation: query in tenant A context returns 0 rows from tenant B
- Token not leaked: sensitive values not present in logs or response bodies
- Scope ceiling: tool with `read` scope cannot execute a `write` operation

### Step 8 — Run and verify

Execute the test suite:
```
pytest src/tests/unit/{service_name}/ -v --cov={target_folder} --cov-report=term
pytest src/tests/integration/{service_name}/ -v
```

Confirm:
- All tests pass
- Coverage ≥ 80% on `{target_folder}`
- No test passes vacuously (comment out the code it tests — it must fail)

- [ ] Every acceptance criterion has a corresponding test
- [ ] Every test actually fails when the code it tests is removed
- [ ] Unit test coverage ≥ 80% on all new `domain/` code
- [ ] Integration tests use Testcontainers — no mocked infrastructure
- [ ] Cross-tenant isolation verified if story touches data
- [ ] Idempotency verified if story involves agents or tools

- [ ] No test uses `assert True` or other vacuous assertions

---

## Forbidden Actions

The AI executing this command must NEVER:

- Write tests that always pass regardless of implementation (vacuous tests)
- Mock the service under test in integration tests
- Skip acceptance criteria tests because they are "covered by unit tests"
- Generate tests without reading the implementation first
- Write tests that depend on external network resources (all infrastructure via Testcontainers)
- Assert on log output as a substitute for real behavioural assertions
- Write a test file without actually running the tests
- Accept < 80% coverage and describe it as "sufficient"
- Modify implementation code to make failing tests pass instead of fixing the tests
- Skip idempotency tests for agent and tool operations
